# 날자 아카이브 블로그 설계

작성일: 2026-06-15  
저장소: `YunyCho/naljabooks-blog`  
1차 배포 주소: `https://yunycho.github.io/naljabooks-blog/`  
최종 배포 주소: `https://blog.naljabooks.com`

## 1. 목적

날자 아카이브는 도서출판 날자와 날자꾸러미가 축적한 연구, 교육 관점, 사례, 정책 해설을 공개하는 전문 글 아카이브다. 보호자, 특수교사, 복지기관 담당자가 전문용어에 막히지 않고 내용을 이해할 수 있어야 하며, 검색엔진과 AI 검색이 출처와 작성 주체를 정확히 파악할 수 있어야 한다.

제품의 공식 정보와 운영 주체의 대표 페이지는 `naljabooks.com`에 둔다. 블로그는 메인 사이트를 복제하지 않고 연구 근거, 사례, 활용 방법과 FAQ 심화 해설을 제공한다.

## 2. 성공 기준

첫 배포는 다음 조건을 모두 충족하면 완료한다.

1. 공개 GitHub 저장소 `YunyCho/naljabooks-blog`가 생성되어 있다.
2. Jekyll 프로덕션 빌드가 오류 없이 완료된다.
3. GitHub Pages 기본 주소에서 홈, 소개, 자료 원칙, 질문 안내, 첫 글이 정상 표시된다.
4. 데스크톱과 모바일에서 읽기 쉽고 주요 탐색 링크가 동작한다.
5. 각 페이지에 고유한 제목, 설명, canonical과 공유 메타데이터가 있다.
6. sitemap, RSS feed, robots.txt가 생성되고 올바른 URL을 사용한다.
7. 첫 글의 연구 주장은 확인 가능한 1차 자료 또는 공신력 있는 원문 출처와 연결된다.
8. 미확인 사실, placeholder, 깨진 링크가 없다.

사용자 정의 도메인 `blog.naljabooks.com` 연결은 GitHub Pages 기본 주소 검증 후 별도 단계로 진행한다.

## 3. 사용자와 문체

### 주요 독자

- 발달장애인 또는 지적장애인의 보호자
- 특수교사와 교육 실무자
- 복지기관 및 관련 단체 담당자

세 집단을 함께 대상으로 하되, 전문 지식이 없는 독자도 이해할 수 있는 표현을 우선한다.

### 작성 원칙

- 첫 2~3문장에서 글의 핵심 답을 제시한다.
- 전문용어는 처음 등장할 때 일상 언어로 풀어서 설명한다.
- 일반 교육 정보와 날자꾸러미의 관점 또는 제품 설명을 구분한다.
- 수치와 연구 주장은 원문 출처와 발행연도를 표시한다.
- 의학적·교육적 효과를 보장하지 않는다.
- 발달장애 청소년과 성인을 아동화하지 않고 학습하고 선택하는 주체로 표현한다.
- 작성일, 수정일, 저자와 정정 방법을 공개한다.

기본 저자 표기는 `도서출판 날자 · 날자꾸러미 편집부`로 한다.

## 4. 기술 선택

### 채택안

GitHub Pages 호환 Jekyll과 자체 제작한 최소 테마를 사용한다.

### 선택 이유

- Markdown 파일 추가만으로 글을 발행할 수 있다.
- 현재 날자꾸러미 브랜드의 색상과 분위기를 필요한 만큼만 계승할 수 있다.
- 불필요한 테마 코드 없이 접근성, 성능, SEO와 구조화 데이터를 통제할 수 있다.
- GitHub Pages의 무료 정적 호스팅과 사용자 정의 도메인을 사용할 수 있다.

### 제외안

- 공개 Jekyll 테마 수정: 시작은 빠르지만 불필요한 코드와 디자인 제약이 생긴다.
- GitHub 기본 테마: 관리하기 쉽지만 브랜드 연결감과 정보 구조 확장성이 부족하다.

## 5. 정보 구조

### 홈

- 블로그 이름 `날자 아카이브`
- 한 문장 소개
- `naljabooks.com`으로 이동하는 명확한 링크
- 최신 글 목록
- 초기 주제 분류
- 편집 및 출처 원칙 안내

### 소개

- 블로그의 목적
- 기본 저자 소개
- 도서출판 날자 및 날자꾸러미 공식 정보는 메인 사이트에서 확인한다는 안내
- 메인 사이트 링크

### 자료 원칙

- 출처 선택 기준
- 연구와 홍보 문구의 구분
- 수정 및 정정 절차
- 개인정보와 사례 사용 원칙

### 질문 안내

- 메인 사이트 FAQ 링크
- 질문별 심화 글 링크
- 중복된 제품 FAQ 전문을 별도 canonical 원문처럼 운영하지 않는다.

### 글 상세

- 제목과 설명
- 작성일과 수정일
- 저자
- 예상 읽기 시간
- 본문과 목차
- 출처 목록
- 관련 글
- 메인 사이트 또는 관련 페이지로 이어지는 다음 행동

### 첫 글

제목은 `지적장애인에게 왜 유추력이 필요할까?`로 한다.

구성:

1. 유추력의 쉬운 정의
2. 유추력이 일상 학습과 선택에 연결되는 방식
3. 지적장애 학습에서 고려할 점
4. 유추력 학습을 설계할 때 지켜야 할 원칙
5. 날자꾸러미가 이 관점을 적용하는 방식과 일반 정보의 구분
6. 핵심 요약
7. 출처

초안 작성 전 관련 연구와 공식 자료를 조사하며, 확인되지 않은 효과 주장은 포함하지 않는다.

## 7. 데이터와 렌더링

### 글 front matter

```yaml
---
layout: post
title: "지적장애인에게 왜 유추력이 필요할까?"
description: "글의 고유한 요약"
date: 2026-06-15
updated: 2026-06-15
author: "도서출판 날자 · 날자꾸러미 편집부"
category: "유추와 문해력"
tags: ["유추력", "문해력", "지적장애"]
sources: []
---
```

실제 게시물에서는 `sources`에 제목, 발행기관, 연도와 원문 URL이 모두 있는 항목만 추가한다. 첫 글은 출처가 채워진 상태에서만 배포한다.

### 페이지 생성 흐름

1. Markdown과 front matter를 작성한다.
2. Jekyll이 공통 레이아웃을 적용해 정적 HTML을 만든다.
3. 홈은 `site.posts`에서 최신 글을 표시한다.
4. `jekyll-sitemap`과 `jekyll-feed`가 sitemap과 RSS를 생성한다.
5. GitHub Pages가 빌드 결과를 공개한다.

## 8. 시각 설계

### 방향

현재 날자꾸러미의 따뜻하고 친근한 분위기를 계승한 편집형 블로그로 만든다.

- 크림색 배경
- 날자꾸러미와 연결되는 주황색 강조색
- 짙은 갈색 또는 검정에 가까운 본문색
- 둥근 카드와 부드러운 경계선
- 넉넉한 여백과 제한된 본문 폭
- 읽기를 방해하는 애니메이션과 장식 배제

### 반응형

- 모바일 우선 CSS를 사용한다.
- 본문은 작은 화면에서 한 열로 표시한다.
- 데스크톱에서는 최대 본문 폭을 제한해 긴 줄을 방지한다.
- 카드 목록은 화면 폭에 따라 한 열 또는 두 열로 변한다.

### 접근성

- 키보드로 모든 링크에 접근할 수 있다.
- 포커스 상태를 분명하게 표시한다.
- 색상만으로 상태나 의미를 전달하지 않는다.
- 본문과 배경의 대비를 충분히 확보한다.
- 장식 이미지는 빈 alt, 정보 이미지는 구체적인 한국어 alt를 사용한다.
- 제목 수준을 순서대로 사용하고 `main`, `nav`, `footer` 등 의미 요소를 적용한다.

## 9. 검색 및 메타데이터

### 플러그인

- `jekyll-seo-tag`
- `jekyll-sitemap`
- `jekyll-feed`

GitHub Pages가 지원하는 버전 범위 안에서 사용한다.

### 페이지 메타데이터

- 고유한 title과 description
- canonical URL
- Open Graph와 Twitter 카드
- 한국어 문서 언어
- 대표 이미지 또는 기본 공유 이미지

### 구조화 데이터

- 글 상세에 `BlogPosting` JSON-LD를 제공한다.
- 작성 주체는 `도서출판 날자 · 날자꾸러미 편집부`로 표시한다.
- 구조화 데이터의 제목, 설명, 날짜와 저자 정보는 화면에 보이는 내용과 일치해야 한다.

### 크롤러 파일

- `robots.txt`: 공개 콘텐츠 접근 허용과 sitemap 위치 제공
- `sitemap.xml`: 플러그인 자동 생성
- `feed.xml`: RSS 자동 생성
- `llms.txt`: 블로그 목적과 핵심 페이지를 안내하는 선택적 보조 파일. 검색 노출 보장 수단으로 설명하지 않는다.

## 10. URL과 배포

### 1차 배포

GitHub Pages 프로젝트 사이트로 배포한다.

```yaml
url: "https://yunycho.github.io"
baseurl: "/naljabooks-blog"
```

모든 내부 링크와 정적 자산 경로는 `relative_url` 또는 `absolute_url` 필터를 사용해 프로젝트 하위 경로에서도 동작해야 한다.

### 최종 도메인 전환

GitHub Pages 기본 주소 검증 후 다음을 진행한다.

1. Gabia에서 `blog` CNAME을 GitHub Pages 대상에 연결한다.
2. GitHub Pages 설정에서 `blog.naljabooks.com`을 사용자 정의 도메인으로 등록한다.
3. HTTPS가 발급되고 강제 적용되는지 확인한다.
4. `_config.yml`을 다음처럼 변경한다.

```yaml
url: "https://blog.naljabooks.com"
baseurl: ""
```

5. canonical, sitemap, feed와 내부 링크를 다시 검증한다.

## 11. 오류 처리와 운영 안전

- 필수 front matter가 빠진 글은 빌드 또는 검증 단계에서 발견한다.
- 외부 출처 URL이 없거나 placeholder인 글은 게시하지 않는다.
- GitHub Pages 빌드가 실패하면 배포 상태를 성공으로 간주하지 않는다.
- 사용자 정의 도메인 전환 전 기본 Pages 주소에서 전체 검증을 완료한다.
- 날짜, 제품 정보 또는 출처가 바뀌면 `updated`를 갱신한다.
- 잘못된 정보 신고를 위한 연락 경로와 정정 원칙을 자료 원칙 페이지에 명시한다.

## 12. 검증 계획

### 자동 검증

- `bundle exec jekyll build --trace` 프로덕션 빌드
- 생성된 홈과 글 HTML 존재 확인
- title, description, canonical, `BlogPosting` JSON-LD 확인
- sitemap.xml, feed.xml, robots.txt 생성 확인
- 내부 링크 및 자산 경로의 `baseurl` 적용 확인
- `TBD`, `TODO`, 임시 URL 등 placeholder 검사

### 브라우저 검증

- GitHub Pages 기본 주소에서 홈과 모든 초기 페이지 접근
- 데스크톱과 모바일 레이아웃 확인
- 키보드 탐색과 포커스 상태 확인
- 메인 사이트, 출처, 관련 글 링크 확인
- 공유 메타데이터와 구조화 데이터 확인
