# 네이버 블로그 초안 자동화 설계

## 목표

GitHub 블로그의 `main` 브랜치에 새 한국어 글이 추가되면 네이버 블로그에 복사해 게시할 수 있는 초안을 자동 생성한다. GitHub의 `_posts` 파일을 유일한 원본으로 유지하고, 네이버 게시 여부와 최종 내용은 사용자가 직접 결정한다.

## 배경과 제약

- 이 저장소는 Jekyll의 `_posts/*.md`를 글 원본으로 사용한다.
- 네이버는 외부 서비스가 블로그 글을 등록하던 공식 글쓰기 API를 2020년 5월 6일 종료했다. 따라서 네이버 임시저장이나 게시를 API로 자동화하지 않는다.
- 브라우저 조작으로 네이버 에디터를 자동화하면 로그인 정보 관리, 화면 변경, 이용 정책에 취약해진다. 이 설계의 범위에서 제외한다.
- OpenAI를 비롯한 외부 AI API를 호출하지 않는다. API 키와 사용료 없이 결정적인 규칙으로 변환한다.
- 영어 글은 대상에서 제외한다.

관련 근거: [네이버 개발자센터 블로그 오픈 API 종료 안내](https://developers.naver.com/notice/article/7527)

## 선택한 접근법

GitHub Actions가 새 원문을 감지하고 `naver-drafts/` 아래에 초안 파일을 만든 뒤 자동 Pull Request를 연다.

다른 선택지로 GitHub Issue나 Actions artifact에 초안을 남길 수 있지만 채택하지 않는다. PR은 초안 수정 이력, 댓글, 검수 상태, 원문과의 연결을 저장소 안에서 함께 관리할 수 있고 artifact처럼 만료되지 않는다.

## 시스템 경계

### 자동화하는 일

1. `main` 브랜치에 추가된 `_posts/*.md` 파일을 찾는다.
2. 한국어 글인지 판별한다.
3. 원문의 의미를 바꾸지 않는 네이버 복사용 초안을 생성한다.
4. 생성 파일만 담은 브랜치와 Pull Request를 만든다.
5. PR 본문에 대상 원문과 검수 체크리스트를 적는다.

### 자동화하지 않는 일

- 기존 `_posts` 파일 수정
- 문장 재작성, 요약, 새로운 주장 추가
- 네이버 로그인, 임시저장, 예약 발행 또는 공개 게시
- 사용자의 PR 승인이나 병합
- 네이버 게시 완료 여부의 자동 판정

## 구성 요소

### 변경 감지 워크플로

`.github/workflows/create-naver-draft.yml`이 `main` 브랜치의 `_posts/**` 변경이 포함된 push에서 실행된다. 이벤트의 이전 커밋과 현재 커밋을 비교해 상태가 `A`인 Markdown 파일만 변환기에 넘긴다. 수정되거나 삭제된 기존 글은 새 글로 간주하지 않는다.

대상 파일이 없으면 성공으로 종료한다. 같은 push에 새 한국어 글이 여러 개 있으면 한 PR에 함께 담는다. 동시 실행은 workflow concurrency로 직렬화한다.

워크플로는 저장소 기본 `GITHUB_TOKEN`만 사용한다. 권한은 `contents: write`와 `pull-requests: write`로 한정하며 별도 서비스 비밀키는 사용하지 않는다.

### 초안 변환기

Ruby 스크립트가 다음 책임만 가진다.

- strict front matter 파싱
- 언어 판별
- 출력 경로 결정과 기존 초안 충돌 확인
- Markdown을 복사 친화적인 텍스트로 변환
- 추적 메타데이터와 복사 영역 작성

언어 값이 없거나 `ko`, `ko-KR`이면 한국어로 취급한다. `en` 등 그 밖의 명시된 언어는 건너뛴다. 현재 저장소가 Ruby/Jekyll을 사용하므로 변환기와 테스트도 Ruby 표준 라이브러리를 우선 사용한다.

### Pull Request 생성기

변환 결과가 하나 이상일 때만 `automation/naver-draft-<짧은 커밋 SHA>` 브랜치를 만들고 커밋·push한다. GitHub CLI로 PR을 생성하며 제목에 원문 글 제목 또는 생성 건수를 표시한다.

PR 본문에는 다음을 포함한다.

- 원문 파일과 공개 URL 목록
- 생성된 초안 파일 목록
- 제목, 문단, 링크, 이미지, 태그 검수 체크리스트
- 이 PR은 네이버 게시를 실행하지 않는다는 안내

## 초안 파일 형식

출력 경로는 `naver-drafts/<원문 파일명에서 날짜 접두사를 제외한 slug>.md`로 한다. 파일은 검수용 메타데이터와 복사 영역을 분리한다.

```markdown
---
source_path: "_posts/2026-06-19-example.md"
source_url: "https://yunycho.github.io/naljabooks-blog/archive/example/"
source_commit: "전체 커밋 SHA"
generated_at: "ISO 8601 UTC 시각"
naver_status: "draft"
---

# 네이버 블로그 복사 영역

## 제목

원문 제목

## 본문

변환된 본문

원문 보기: https://yunycho.github.io/naljabooks-blog/archive/example/

## 해시태그

#첫번째태그 #두번째태그
```

`source_path`와 `source_commit`은 어떤 원문으로부터 생성되었는지 추적한다. `naver_status`는 사람을 위한 기록이며 자동 게시 신호로 사용하지 않는다.

## 변환 규칙

변환기는 내용을 재창작하지 않고 표현 형식만 바꾼다.

1. YAML front matter를 본문에서 제거한다.
2. 제목은 front matter의 `title`을 그대로 사용한다.
3. Markdown 제목의 목차 앵커 `{#section-id}`를 제거한다.
4. 인라인 링크 `[문구](URL)`를 `문구 (URL)`로 바꾼다.
5. 이미지는 `이미지: 대체 텍스트 (절대 URL)`로 바꾼다. 저장소 상대 이미지 경로는 블로그 공개 URL 기준의 절대 URL로 만든다.
6. 강조 표식은 벗기되 안의 문구는 유지한다.
7. 목록과 인용문은 읽을 수 있는 일반 텍스트 구조를 유지한다.
8. 연속된 과도한 빈 줄을 정리한다.
9. 본문 마지막에 GitHub 원문 URL을 추가한다.
10. front matter의 `tags`를 공백 없는 네이버 해시태그로 바꾼다. 태그 내부 공백과 해시태그에 쓸 수 없는 구두점은 제거한다.

변환 결과에 없는 정보를 추론하거나 보충하지 않는다.

## 중복과 원본 보호

- 변환기는 입력 경로가 `_posts/` 아래인지 확인한다.
- 출력은 `naver-drafts/` 아래에서만 허용한다.
- 대상 출력 파일이 기준 브랜치에 이미 있으면 덮어쓰지 않고 건너뛴다.
- 워크플로의 커밋 대상은 생성된 `naver-drafts/` 파일로 제한한다.
- PR 브랜치는 원문을 추가한 `main` 커밋에서 시작한다.

이 제약을 통해 자동화가 원문을 편집하거나 같은 초안을 반복 생성하지 못하게 한다.

## 오류 처리

다음 경우 변환을 실패시키고 PR을 만들지 않는다.

- front matter 구분자나 YAML이 잘못됨
- 필수 `title`이 없거나 빈 값임
- 입력 또는 출력 경로가 허용 범위를 벗어남
- 출력 파일 쓰기 실패
- Git commit, push 또는 PR 생성 실패

영어 글, 새 글이 아닌 변경, 이미 초안이 존재하는 글은 오류가 아니라 명시적인 skip 결과다. Actions 로그에는 생성·건너뜀·실패 대상을 구분해 남긴다.

한 push에 여러 대상이 있고 하나가 잘못된 경우 전체 실행을 실패시킨다. 일부 초안만 PR로 올려 누락을 숨기지 않는다.

## 테스트 전략

Ruby의 Minitest로 변환기를 테스트한다.

- 언어 값 없음, `ko`, `ko-KR`은 처리하고 `en`은 건너뜀
- front matter 제거와 제목 보존
- 제목 앵커 제거
- 링크와 이미지 URL 변환
- 강조, 목록, 인용문의 텍스트 보존
- 원문 URL 생성
- 태그의 네이버 해시태그 변환
- 기존 출력 파일 중복 방지
- 잘못된 YAML, 누락된 제목, 경로 이탈 거부
- 현재 저장소의 실제 한국어 글 fixture를 사용한 통합 변환
- 여러 입력 중 하나가 실패할 때 출력이 커밋 가능한 상태로 남지 않음

워크플로 자체는 변환기 테스트를 먼저 실행한 뒤에만 초안 생성과 PR 생성을 진행한다. 로컬에서도 같은 테스트 명령을 실행할 수 있게 README에 기록한다.

## 운영 절차

### 최초 설정

1. 저장소 Actions가 workflow를 실행할 수 있게 한다.
2. Workflow permissions에서 `GITHUB_TOKEN`의 read/write 권한과 Pull Request 생성을 허용한다.
3. 구현 PR을 병합한 뒤 테스트용 한국어 글 또는 수동 검증 절차로 자동 PR 생성을 확인한다.

### 글마다 반복되는 절차

1. 사용자가 `_posts`에 한국어 글을 추가해 `main`에 병합하거나 push한다.
2. Actions가 네이버 초안 PR을 연다.
3. 사용자가 PR에서 초안의 제목, 본문, 링크, 이미지, 태그를 검수하고 필요하면 수정한다.
4. 사용자가 PR을 병합해 검수한 초안을 저장소에 기록한다.
5. 사용자가 초안의 복사 영역을 네이버 에디터에 붙여 넣는다.
6. 네이버 미리보기에서 레이아웃과 링크를 마지막으로 확인한 뒤 직접 게시한다.

## 완료 조건

- `main`에 새 한국어 post가 추가되면 별도 비밀키 없이 초안 PR이 자동 생성된다.
- 영어 post, 기존 post 수정, post 외 파일 변경에는 초안 PR이 생성되지 않는다.
- 생성된 파일에서 제목, 본문, 원문 링크, 해시태그를 복사할 수 있다.
- 자동화는 `_posts`를 수정하지 않고 네이버에 로그인하거나 게시하지 않는다.
- 중복 초안과 잘못된 원문은 명확하게 건너뛰거나 실패한다.
- 로컬과 Actions에서 변환기 테스트가 통과한다.
